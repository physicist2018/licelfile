# AGENTS.md: Правила для AI‑агентов

**Проект:** бэкенд на Go (гексагональная архитектура, роутер chi)  


## Цель

Обеспечить единообразное поведение AI‑агентов при генерации и рефакторинге кода. Агент должен понимать стек технологий и архитектурный стиль проекта.

## Архитектурные принципы

* **Гексагональная (Ports & Adapters) архитектура:**
    * Чёткое разделение на слои: `domain`, `application`, `infrastructure`, `interfaces`.
    * Зависимости направлены внутрь: внешние слои зависят от внутренних, но не наоборот.
    * Порты (интерфейсы) определяются в слое `application`.
    * Адаптеры (реализации) располагаются в слое `infrastructure` (БД) и `interfaces` (HTTP, CLI).
* **Clean Architecture:** соблюдай SRP (принцип единственной ответственности), избегай циклических зависимостей.
* **Контекст (`context.Context`):** всегда передавай `ctx context.Context` в сигнатуры функций, работающих с БД, внешними API или имеющими таймаут.

## Технологический стек

* **Язык:** Go (1.25).
* **Веб‑фреймворк:** `chi` (минималистичный роутер).
* **Работа с БД:** предпочтительно использовать интерфейсы репозиториев (`UserRepository`, `OrderRepository`).
* **Форматирование:** `gofmt`.
* **Линтинг:** `golangci-lint`.

## Конвенции именования

* **Пакеты:** короткие, строчные имена (`user`, `order`, `auth`).
* **Типы и интерфейсы:** CamelCase (`User`, `PaymentService`).
* **Интерфейсы:** обычно суффикс `er` (`Repository`, `Service`, `Validator`).
* **Функции и методы:** CamelCase, глаголы в начале (`CreateUser`, `ValidateEmail`).
* **Переменные:** описательные, CamelCase (`userID`, `paymentAmount`).

## Структура проекта (Canonical Paths)

/cmd/
/web/ # Точка входа веб‑приложения
/internal/
/domain/ # Бизнес‑сущности и доменные события
/application/ # Юзкейсы, порты (интерфейсы)
/infrastructure/# Адаптеры для БД, внешних сервисов
/interfaces/ # Адаптеры для HTTP (роуты, хендлеры), CLI
/pkg/ # Переиспользуемые утилиты вне домена
/configs/ # Конфигурационные файлы
/migrations/ # Миграции БД

## Workflow для типовых задач

### 1. Добавление новой фичи (HTTP‑эндпоинт)

1. Определи контракт (интерфейс) в `/internal/application/ports/`.
2. Реализуй бизнес‑логику в `/internal/application/usecases/`.
3. Создай HTTP‑хендлер в `/internal/interfaces/http/handlers/`, который использует юзкейс.
4. Зарегистрируй эндпоинт в роутере (`/internal/interfaces/http/router.go`) с помощью `chi`.
5. Создай адаптер репозитория в `/internal/infrastructure/repository/`.

### 2. Рефакторинг существующего кода

* Сохраняй существующий контракт (сигнатуры функций, интерфейсы).
* Разбивай большие функции на более мелкие, следуя SRP.
* Выноси дублирующийся код в утилиты (`/pkg/`).

## Запрещённые паттерны

* **Hardcoded‑значения** в бизнес‑логике (строки подключения, API‑ключи). Используй конфигурационный пакет.
* **Прямые вызовы БД** из `interfaces`. Всегда используй репозитории.
* **Игнорирование ошибок.** Обрабатывай все возвращаемые ошибки.
* **Использование `panic` в бизнес‑логике** (кроме крайних случаев в `main`).

## Инструкции для агента (DeepSeek V4)

При выполнении любых задач (генерация кода, рефакторинг, ответы на вопросы):

1. Опирайся на описанные выше архитектурные принципы и структуру проекта.
2. Генерируй код, совместимый с Go и `chi`.
3. Соблюдай конвенции именования.
4. Следуй workflow для типовых задач.
5. Избегай запрещённых паттернов.
6. Пиши лаконично и понятно. Не добавляй избыточных комментариев, если код самодокументируемый.
7. Если не хватает контекста — задай уточняющий вопрос.
8. Перед началом кодирования - выводи план изменений на согласование
