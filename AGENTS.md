# AGENTS.md: Правила для AI‑агентов

**Проект:** бэкенд на Go (чистая архитектура, framework  chi)  


## Цель

Обеспечить единообразное поведение AI‑агентов при генерации и рефакторинге кода. Агент должен понимать стек технологий и архитектурный стиль проекта.

## Архитектурные принципы
* **Clean Architecture:** соблюдай SRP (принцип единственной ответственности), избегай циклических зависимостей.
* **Контекст (`context.Context`):** всегда передавай `ctx context.Context` в сигнатуры функций, работающих с БД, внешними API или имеющими таймаут.

## Технологический стек

* **Язык:** Go (>=1.24).
* **Веб‑фреймворк:** `gin`
* **Работа с БД:** предпочтительно использовать интерфейсы репозиториев (`UserRepository`, `OrderRepository`).
* **Форматирование:** `gofmt`.
* **Линтинг:** `golangci-lint`.

## Конвенции именования

* **Пакеты:** короткие, строчные имена (`user`, `order`, `auth`).
* **Типы и интерфейсы:** CamelCase (`User`, `PaymentService`).
* **Интерфейсы:** обычно суффикс `er` (`Repository`, `Service`, `Validator`).
* **Функции и методы:** CamelCase, глаголы в начале (`CreateUser`, `ValidateEmail`).
* **Переменные:** описательные, CamelCase (`userID`, `paymentAmount`).

## Структура проекта (Canonical Paths)

my-go-app/
├── cmd/                  # Точка входа приложения
├── internal/
│   ├── domain/         # Entities: бизнес‑модели
│   ├── usecase/        # Use Cases: бизнес‑логика
│   ├── repository/       # Репозитории (интерфейсы + реализации)
│   │   └── postgres/     # Реализация для PostgreSQL
│   ├── delivery/       # Адаптеры для внешних запросов
│   │   ├── http/       # HTTP‑обработчики (Gin, Echo и т. д.)
│   └── config/         # Конфигурация приложения
├── pkg/                # Переиспользуемые пакеты (вне бизнес‑логики)
├── migrations/         # Миграции БД
├── docs/             # Документация API (Swagger и т. д.)
├── tests/            # Интеграционные тесты
├── go.mod
└── README.md

## Важные принципы
1. Зависимости внутрь:
    - внутренние слои (domain, usecase) не импортируют внешние;
    - интерфейсы репозиториев определяются в usecase или repository, их реализации — во внешних слоях.

2. Инверсия зависимостей:
    - Use Cases зависят от абстракций (интерфейсов репозиториев), а не от реализаций.

3. Изоляция бизнес‑логики:
    - домен и сценарии использования не знают о БД, HTTP, фреймворках;
    - изменения в БД или API не затрагивают бизнес‑правила.

4. Тестируемость:
    - Use Cases легко тестировать моками репозиториев;
    - Delivery‑слой тестируется с моками Use Cases.
## Workflow для типовых задач

### 1. Добавление новой фичи (HTTP‑эндпоинт)

1. Определите затронутые доменные сущности (internal/domain/). При необходимости создайте новые.
2. Опишите Use Cases (бизнес‑сценарии) и соответствующие методы.
3. Определите требуемые интерфейсы репозиториев.
4. Продумайте HTTP‑эндпоинт (путь, метод, структура запроса/ответа).
5. Составьте чек‑лист для тестирования.
5. Доменные модели: создайте/обновите структуры в internal/domain/.
6. Use Cases: реализуйте бизнес‑логику в internal/usecase/
  - определите интерфейсы Use Case‑сервисов;
  - реализуйте методы с соблюдением SRP;
  - всегда передавайте ctx context.Context в сигнатуры функций, работающих с БД/API/таймаутами.
7. Репозитории: создайте реализацию интерфейсов в internal/repository/postgres/:
  - используй GORM
  - соблюдайте инверсию зависимостей (Use Case зависит от интерфейса).
8. HTTP‑обработчики: в internal/delivery/http/ реализуйте:
  - парсинг и валидацию входящих данных;
  - вызов методов Use Case с ctx;
  - формирование HTTP‑ответа (статус, тело).
9. Роутинг: зарегистрируйте эндпоинт в роутере Gin в cmd/.
10. Тестирование
  - Напишите юнит‑тесты для Use Cases с моками репозиториев (в internal/usecase/ или tests/unit/):
  - проверьте позитивные и негативные сценарии;
  - протестируйте обработку ctx (отмена, таймаут).
11. При необходимости обновите README.md, CHANGELOG.md
12. 

 

### 2. Рефакторинг существующего кода

* Сохраняй существующий контракт (сигнатуры функций, интерфейсы).
* Разбивай большие функции на более мелкие, следуя SRP.
* Выноси дублирующийся код в утилиты (`/pkg/`).

## Запрещённые паттерны

* **Hardcoded‑значения** в бизнес‑логике (строки подключения, API‑ключи). Используй конфигурационный пакет.
* **Прямые вызовы БД** из `interfaces`. Всегда используй репозитории.
* **Игнорирование ошибок.** Обрабатывай все возвращаемые ошибки.
* **Использование `panic` в бизнес‑логике** (кроме крайних случаев в `main`).

## Инструкции для агента (DeepSeek V4)

При выполнении любых задач (генерация кода, рефакторинг, ответы на вопросы):

1. Опирайся на описанные выше архитектурные принципы и структуру проекта.
2. Генерируй код, совместимый с Go и `gin`.
3. Соблюдай конвенции именования.
4. Следуй workflow для типовых задач.
5. Избегай запрещённых паттернов.
6. Пиши лаконично и понятно. Не добавляй избыточных комментариев, если код самодокументируемый.
7. Если не хватает контекста — задай уточняющий вопрос.
8. Перед началом кодирования - выводи план изменений на согласование
